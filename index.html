QueryDSL

동적 쿼리를 타입 세이프하게 — JPA 메서드 쿼리의 한계를 넘어서

왜 QueryDSL인가? Spring Data JPA의 메서드 쿼리 방식(findByNameAndStatusAndPriceBetween...)은 조건이 늘어날수록 한계에 부딪힙니다. JPQL 문자열은 오타를 컴파일러가 잡지 못하고, 조건이 있을 때만 WHERE절을 붙이는 동적 쿼리 작성도 어렵습니다. QueryDSL은 Q타입을 통해 코드로 쿼리를 조립하므로, 오타는 컴파일 에러로 즉시 발견되고 조건 조합도 메서드 체이닝으로 명확해집니다.
왜 QueryDSL인가?

❌ JPQL 문자열의 한계

// 런타임에야 에러 발견
"SELECT u FROM User u"
"WHERE u.nAme = :name"
// ↑ 오타를 컴파일러가 못 잡음

✅ QueryDSL 타입 세이프

QUser user = QUser.user;
queryFactory
  .select(user)
  .where(user.name.eq("kim"))
// ↑ 컴파일 타임에 오타 차단
동적 쿼리 빌더 — 조건을 켜고 끄며 쿼리 변화 확인
BooleanExpression의 핵심: 각 조건 메서드가 null을 반환하면 QueryDSL이 자동으로 해당 조건을 무시합니다. if문 없이 "값이 있을 때만 조건 추가"가 가능합니다. 아래에서 조건을 켜고 끄면서 실제 생성되는 코드 변화를 확인해 보세요.

N+1 문제 — 조회 전략 비교
N+1 문제란? 주문 10건을 조회한 뒤(쿼리 1번), 각 주문의 회원 정보를 개별로 가져올 때 주문마다 별도 쿼리가 실행됩니다(쿼리 N번). 결국 1 + N개의 쿼리가 발생합니다. 데이터가 100만 건이면 100만+1번의 쿼리가 나갑니다.

Index & 쿼리 튜닝

실행 계획으로 재현 가능하게 — 감이 아니라 데이터로 개선한다

인덱스란? 책의 "색인(찾아보기)"과 같습니다. 인덱스가 없으면 DB는 테이블 전체를 처음부터 끝까지 스캔(Full Table Scan)합니다. 인덱스가 있으면 B-Tree를 타고 O(log N)으로 빠르게 원하는 행을 찾습니다. 단, 인덱스는 추가 저장공간을 차지하고 INSERT/UPDATE/DELETE 시 인덱스도 함께 갱신해야 해서 쓰기 성능이 약간 저하됩니다.
B-Tree 인덱스 — 값을 입력하고 탐색 과정 확인
B-Tree 탐색 원리: 루트 노드부터 시작해 찾는 값과 노드의 키를 비교하며 왼쪽/오른쪽 자식으로 내려갑니다. 트리의 높이가 3이면 최대 3번의 비교만으로 수백만 건 중 원하는 데이터를 찾을 수 있습니다. 1~100 사이 값을 입력해 탐색 경로를 확인해 보세요.
값을 입력하고 탐색 버튼을 누르면 단계별 탐색 과정이 표시됩니다.

동시성 제어

재고 차감, 포인트 사용 — 동시에 10명이 누르면 무슨 일이 생길까?

동시성 문제란? 재고 1개인 상품에 두 명이 동시에 주문합니다. 두 스레드가 동시에 재고를 읽으면 둘 다 "1개 있음"을 보고 차감 후 저장합니다. 결과적으로 재고가 -1이 되거나(Dirty Write), 한 명의 차감이 덮어써지는(Lost Update) 현상이 발생합니다.

분산 락 — Redisson을 이용한 구현
Redisson
RLock lock = redissonClient.getLock("stock:product:1");

try {
  // 최대 5초 대기, 3초 후 자동 해제
  boolean acquired = lock.tryLock(5, 3, TimeUnit.SECONDS);
  if (!acquired) throw new LockException();

  // 재고 차감 로직
  stockService.decrease(productId, quantity);
} finally {
  if (lock.isHeldByCurrentThread()) lock.unlock();
}
💡 언제 분산 락이 필요한가?
서버가 여러 대일 때 — DB 락은 같은 DB에 연결된 경우만 효과가 있습니다. 여러 서버가 같은 Redis를 바라보게 해서 서버 경계를 넘는 락을 구현합니다.
